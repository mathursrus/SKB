# Coaching Moment: shipped-unindexed-query-to-prod

## What happened

On 2026-04-13, I shipped commit c39842d as a UX refinement on top of the dynamic-ETA feature (7e67fd0). The refinement changed `getEffectiveTurnTime` to always call `computeDynamicTurnTime` regardless of mode, so the UI could know whether the Dynamic option should be visible. The `computeDynamicTurnTime` query is `queueEntries.find({ locationId, state: 'departed', seatedAt: {$exists}, departedAt: {$exists} }).sort({ departedAt: -1 }).limit(20)` — and that query was not backed by any index. The only relevant index on the collection was `loc_serviceDay_state_joinedAt`, which could not serve the query because there is no serviceDay filter and departedAt isn't in the index at all. Local dev databases (including `skb_dev_validate` and the integration-test db) had too little data for the collection-scan + in-memory-sort to hit Mongo's 32MB sort limit, so local tests passed and I pushed to master. Azure auto-deployed and within ~90 seconds every endpoint that resolved an ETA — `/api/queue/state`, `/api/queue/join`, `/api/queue/status`, `/api/host/queue`, `/api/host/stats`, `/api/host/settings` — started returning 503 "temporarily unavailable" for SKB Bellevue. The diner queue page and the host dashboard were both non-functional for ~15 minutes, from ~16:45 UTC until ~17:03 UTC. I only discovered the outage because the user asked "have you validated in prod?" after I had already claimed the commit was complete; when I ran the actual prod validation they requested, the 503s were immediately visible. My first recovery attempt (commit 7211079) added a try-catch around `computeDynamicTurnTime` and added the missing index to `bootstrapIndexes`. That hot-fix did not recover prod within the 2-3 minute window I gave it, so I reverted both commits (c39842d and 7211079) via git revert. The revert restored prod to the last known-good state: `computeDynamicTurnTime` only runs when mode=dynamic, SKB is in manual mode, the broken path is never exercised. Verified healthy via Playwright after deploy 24356111805 at 17:03 UTC.

## What was learned

The L1 mistake-patterns entry "Evidence-based reasoning required for external-dependent work and load-bearing numbers" (promoted today, commit d4318c9) applies not just to strategy-document numbers but to any code change that adds a new query against a production-scale collection. The specific missing check: when adding a query, explicitly verify it is backed by an index BEFORE committing — `db.collection.find(...).sort(...).explain('executionStats')` should show an IXSCAN or FETCH stage, never a COLLSCAN + in-memory SORT on a collection with meaningful size. Local dev data is a false-positive signal when the failure mode is data-volume-dependent.

## What the agent should have done

Before committing c39842d, the agent should have asked the exact pre-commit question that the just-synthesized MC-1 L1 entry recommends: "What's the one number or claim or query in this work that would sink the whole thing if it were wrong at production scale?" The answer was "the unindexed departedAt query runs on every ETA resolution and is untested against SKB Bellevue's real data volume." Concrete alternative steps the agent should have taken: (1) run `db.queueEntries.find({ ... }).sort({ departedAt: -1 }).limit(20).explain('executionStats')` against a locally-seeded collection with a few thousand departed parties to confirm the query plan, (2) add the index in a SEPARATE commit that ships FIRST, wait for deploy, verify the index exists, and only then ship the code that depends on it, (3) during the code PR, add a partial index on { locationId, state, departedAt } as part of `bootstrapIndexes` so the query is covered from the first invocation. The agent did none of these. The hot-fix attempt (7211079) was also rushed — it added the index and a try-catch but did not test the index-creation path against a prod-sized collection, and the try-catch was not verified to be reached (the 2-3 minute recovery window was too short to confirm). Finally, the root coaching-level failure: the user had to PROMPT the agent to validate in prod. The MC-2 L1 entry was literally about this: the user reads committed work carefully and catches errors after the fact; the counter-pattern is to insert a pre-commit check by the agent. The agent did not. Two L1 entries synthesized and promoted less than 2 hours before this incident both described the exact failure pattern that caused the incident. That is a signal that knowing the rule is not the same as applying it, and the agent should treat the pre-commit pre-mortem check as a hard gate for any change that (a) adds a database query, (b) touches external APIs, or (c) modifies a path exercised by live production data.

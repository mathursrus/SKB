# OSH iOS — Compliance & Privacy

This document covers App Store Review Guidelines, privacy manifest, accessibility, and export-control decisions for the OSH iOS app. The app is a **restaurant operations tool** for staff (host / admin / owner) plus a guest queue tracker — which materially narrows the compliance surface.

## 1. Data collection inventory

| Data type | Collected? | Purpose | Linked to identity? | Used for tracking? |
|---|---|---|---|---|
| Name (guest) | Rendered only | Display in host queue | No — remains server-side, not persisted on device | No |
| Phone (guest) | Rendered only | `tel:` dial + chat SMS recipient | No — remains server-side | No |
| Host PIN | Yes (device) | Cookie session persisted in `expo-secure-store` → iOS Keychain | No | No |
| Analytics / crash data | Deferred to Phase 7 | Error reporting | No | No |
| IDFA | **Never** | — | — | — |
| Location | **Never** | — | — | — |
| Contacts | **Never** | — | — | — |
| Photos | **Never** | — | — | — |

Because the app renders guest data fetched from the OSH backend and never writes guest PII to device storage, the app's App Store privacy label is expected to be **"Data Not Collected"** from the user (the host). The OSH backend's privacy posture is a separate concern documented in the main repo.

## 2. Privacy manifest (`PrivacyInfo.xcprivacy`)

Generated by `app.json` → `ios.privacyManifests`. Declarations:

- **NSPrivacyTracking:** `false` — no cross-app tracking SDK.
- **NSPrivacyCollectedDataTypes:** `[]` — the app does not collect user-scoped data.
- **NSPrivacyAccessedAPITypes:** declared reasons for required APIs that Apple flags in PrivacyInfo:

| API category | Reason code | Why |
|---|---|---|
| `UserDefaults` | `CA92.1` | Zustand persists app state and `expo-secure-store` writes to UserDefaults during its own bootstrap. Reason CA92.1 = "Access info from same app, per documentation." |
| `FileTimestamp` | `C617.1` | React Native's Metro/Expo bundler touches file timestamps for HMR and cache validation. Reason C617.1 = "Display to the person using the device." |

If Phase 7 adds Sentry or another observability SDK, it may require additional `NSPrivacyAccessedAPITypes` entries — recheck at that time.

## 3. AI disclosure

**Not applicable.** This app contains no AI-generated content, no ML models on-device, and no AI-driven UX. No AI disclosure modal is required.

## 4. TCPA (SMS consent)

Covered by the spec (§4, Issue #30) at the **backend** layer: consent is captured at guest check-in time when the guest enters their phone. The iOS app:
- Does NOT have a path to acquire new SMS consent from within the app.
- Only triggers SMS on parties whose phone was already collected with consent.
- Chat templates are the same ones reviewed by OSH operations; no ad-hoc wording that could violate TCPA tone rules.

## 5. Accessibility (WCAG 2.1 AA + iOS VoiceOver)

Ref spec §3.4 R19. Status per surface:

| Surface | Label | Role | Focusable | Contrast |
|---|---|---|---|---|
| Login PIN input | "Host PIN" | textbox | Yes | AA (text `#f4f4f5` on `#1e222c`) |
| Login button | "Sign in" | button | Yes | AA |
| Waiting row | "Party N, <name>, size S" | summary | Yes | AA |
| Row: Seat | "Seat" | button | Yes | AA (primary amber) |
| Row: Chat | "Chat with <name>" | button | Yes | AA; badge announced via accessibilityValue |
| Row: Call | "Call <name>" | button | Yes | AA |
| Seat dialog: Table # | "Table number" | textbox | Yes, autoFocus | AA; error state bumps border contrast |
| Seat dialog: override | "Override and seat anyway at table N" | button | Yes | AA |
| Chat composer | "Message composer" | textbox | Yes | AA |

Known gaps:
- Hardware-keyboard focus order on iPad is left-to-right, top-to-bottom by default via expo-router — not explicitly tested yet.
- Dynamic Type (iOS font scaling) is not overridden and RN styles use fixed `fontSize` — iOS will scale up to the user's preferred size automatically but some row layouts may clip at the largest sizes. Flagged for Phase 9.

## 6. Export compliance

`app.json` → `ios.infoPlist.ITSAppUsesNonExemptEncryption = false`.

Justification: the app uses only HTTPS (standard TLS via iOS URL loading system) and no custom cryptography. Per Apple guidance, apps that use only exempt encryption are not required to submit an annual self-classification report. This is declared at the Info.plist level so App Store Connect doesn't prompt on every upload.

## 8. Content moderation

**Not applicable.** The app renders waitlist entries created by restaurant guests via the OSH public queue form. Content moderation on those names lives in the backend (`tests/unit/nameRedact.test.ts` exists in the main repo) — the iOS client faithfully displays whatever the server sends. Hosts and admins are the only authenticated users of this app, and they sign in with email + password.
